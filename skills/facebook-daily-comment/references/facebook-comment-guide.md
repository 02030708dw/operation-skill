# Facebook 评论与连接排查（MYT Android）

此文档只用于排查。Facebook 登录、验证码和账号安全验证必须由用户本人在云手机画面中完成。

## 推荐验证顺序

1. 使用 `--check` 检查所有目标 MYT 端口。
2. 在云手机画面中确认 Facebook 已登录并能正常加载动态。
3. 使用较小的 `--count` 执行预演。
4. 确认预演识别的是帖子评论按钮。
5. 先对单台设备发送一条测试评论。
6. 人工核对评论成功后，再增加设备或评论数量。

## 常见问题

### MYT API 无法连接

- `MYT_HOST` 只能填写主机名或 IP，不含协议和端口。
- 检查 Hermes 电脑与 MYT 控制器是否网络互通。
- 默认端口为 T1001=10005、T1002=10008，每台递增 3。
- 自定义端口映射时，直接向 `--devices` 传数字端口。

### UI dump 为空或不是 XML

- 新版脚本会对每次 UI dump 自动重试，默认 4 次。
- 默认启动使用 Facebook Feed 深链，不会 `force-stop`，既可从桌面恢复 App，也能避免继续停留在上次的评论详情。
- 只有 App 状态明显异常时才使用 `--force-restart`。一般恢复任务使用默认启动，不要加 `--no-launch`。
- `--timeout` 是单次 HTTP 请求时限，不会让 Facebook 更快完成渲染。

### 找不到评论按钮

- 使用 `--verbose` 查看可点击节点描述。
- 确认当前页面是 Facebook 动态，而不是登录页、通知页或弹窗。
- 默认支持中文简体、中文繁体和英文常见文案。
- 选择器只接受完整的评论按钮标签，防止把“给评论留下心情”等点赞/反应说明误认为评论入口。
- Facebook 更新文案后，应扩展精确标签列表，不要恢复宽松的子字符串匹配，也不要改成固定坐标。

### 当前停留在评论详情或手机桌面

- 评论详情通常包含“写评论”输入框、回复按钮以及评论点赞按钮，但没有 Feed 的精确“评论”入口。
- 脚本会检测评论详情，并最多逐层安全关闭两次；每次关闭后重新 dump UI，不会盲按多次返回键。
- 如果 `--no-launch` 时设备位于桌面或其他 App，脚本会快速返回 `screen-state-error`。
- 补跑时去掉 `--no-launch`，默认 Feed 深链会恢复 Facebook；不要临时编写 HTTP 脚本连续发送返回键，否则可能直接退出 Facebook。

### 找不到输入框或发送按钮

- 等待评论面板完全载入后再试。
- 检查是否出现账号安全提示、网络错误或权限弹窗。
- 脚本会自动轮询等待输入框和发送按钮；使用 `--verbose` 检查每次重试以及可点击、可编辑、可聚焦节点。
- 不要恢复旧版的固定发送坐标；不同分辨率和键盘状态会改变布局。

### 评论内容无法输入

脚本默认只支持可打印 ASCII。中文、Emoji 或某些特殊字符需要云手机安装支持 Unicode 的 ADB 输入法，并对脚本增加专用输入通道。

不要删除文本校验后直接拼接 shell 命令，否则可能造成命令注入或错误输入。

### 评论数量不足

- 页面可能没有足够的普通帖子。
- 评论面板可能被弹窗或网络错误阻挡。
- 先减少 `--count` 并观察单次行为。
- 增加 `--max-cycles` 前先阅读详细日志。
- 人工核对是否出现重复帖子或 Facebook 风控提示。

### 部分完成或 Hermes 显示 exit 124

- 新版脚本默认设置每台设备 105 秒内部时限，应在常见的 120 秒外层超时前输出完整或部分汇总。
- `--timeout` 只控制单个 HTTP 请求，不是脚本总运行时间；不要靠增大它解决总超时。
- 若日志已经显示 `send tap issued (3/3)`，表示发送点击已发出。不要直接重跑整批任务，以免重复评论。
- 汇总会显示 `commented=2/3 remaining=1` 并输出 `RESULT_JSON`。
- 对部分完成设备，只能使用剩余数量补充；不能再次使用原始 `--count 3`。默认不要添加 `--no-launch`。
- 已完成的其他设备不得加入补跑命令。若剩余数量不确定，停止操作并让用户人工核对。
- 为诊断而重跑时必须移除 `--execute`；`--verbose --execute` 仍会真实发送评论，不是只读诊断。
